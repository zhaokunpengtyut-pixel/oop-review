## 一、选择题答案与详细解析

### ● 绪论与基本概念（第1.1-1.8题）

---

#### 1.1 面向过程程序设计关注的是：**B. 怎么做**

| 选项 | 分析 |
|:----:|:------|
| A. 谁来做 | ❌ 这是**面向对象**的关注点——把任务分配给不同对象，对象间通过消息协作 |
| **B. 怎么做** | ✅ **正确。** 面向过程编程（如C语言）关注解决问题的步骤和流程，像写菜谱一样一步步列出操作。核心是"函数"，数据与操作分离 |
| C. 对象之间的关系 | ❌ 这是面向对象的概念，描述对象之间的协作方式 |
| D. 数据封装 | ❌ 这是面向对象的特征之一（封装性），不是面向过程的关注点 |

**📌 考点扩展：**
- **面向过程** = 以"函数"为中心，自顶向下、逐步求精、模块分解、分而治之
- **面向对象** = 以"对象"为中心，抽象、封装、继承、多态
- 代表语言对比：C（面向过程） vs C++/Java/Python（面向对象）

---

#### 1.2 以下哪项不是面向对象程序设计的基本特征？**C. 递归性**

| 选项 | 分析 |
|:----:|:------|
| A. 抽象性 | ✅ 是基本特征。对同类对象的共同属性和行为进行概括，抽出共性形成类 |
| B. 封装性 | ✅ 是基本特征。将数据和操作数据的方法封装在一起，对外只暴露简单接口 |
| **C. 递归性** | ❌ **不是。** 递归是一种编程技术（函数调用自身），不是面向对象的特征 |
| D. 多态性 | ✅ 是基本特征。同一消息被不同类型对象接收时产生不同行为 |

**📌 考点扩展：**
面向对象的四大基本特征是：**抽象性、封装性、继承性、多态性**（注意：有些教材把继承和派生合在一起算一个特征，所以是四大特征）。
- 抽象是基础→封装是手段→继承是复用→多态是升华

---

#### 1.3 类与对象的关系，最恰当的比喻是：**B. 蓝图与房屋**

| 选项 | 分析 |
|:----:|:------|
| A. 汽车与轮胎 | ❌ 这是**整体与部分**的关系（聚合），不是类与对象。轮胎是汽车的一个组成部分 |
| **B. 蓝图与房屋** | ✅ **最恰当。** 蓝图定义了房屋的结构和规格（类），按蓝图建成的具体房屋就是对象。一张蓝图可以建造多栋房屋（一个类可以创建多个对象） |
| C. 树干与树枝 | ❌ 这是整体与部分的关系，且是树状结构的一部分 |
| D. 河流与水滴 | ❌ 河流是水滴的集合，但关系不太精确 |

**📌 考点扩展：**
经典比喻：**模具与铸件**（更常见）。类 = 模板/设计图，对象 = 按照模板生产的真实产品。
- 类不占内存，对象占内存
- 一个类可以创建**多个**对象
- 每个对象有自己独立的存储空间（静态成员除外）

---

#### 1.4 "一个接口，多种状态"描述的是：**C. 多态**

| 选项 | 分析 |
|:----:|:------|
| A. 继承 | ❌ 继承是子类自动获得父类属性和行为，目的是代码重用，不是"一个接口多种状态" |
| B. 封装 | ❌ 封装是将数据和操作捆绑在一起对外隐藏细节，目的是信息隐蔽 |
| **C. 多态** | ✅ **正确。** "一个接口，多种状态"是多态性的经典定义。同样的函数调用（接口），在不同对象上执行不同的行为 |
| D. 抽象 | ❌ 抽象是抽出共性形成类的过程 |

**📌 考点扩展：**
多态的分类（四种）：
1. **重载多态**（函数重载、运算符重载）— 编译时 / 静态
2. **强制多态**（类型转换）— 编译时 / 静态
3. **包含多态**（虚函数）— 运行时 / 动态 ⭐ 最重要
4. **参数多态**（模板）— 编译时 / 静态

---

#### 1.5 下面关于消息的描述，错误的是：**D. 对象必须对收到的消息做出响应**

| 选项 | 分析 |
|:----:|:------|
| A. 同一对象可接收不同形式的多个消息 | ✅ 正确。一个对象可以有多个不同的成员函数，可以接收不同形式的消息 |
| B. 相同形式的消息发送给不同对象，响应可以截然不同 | ✅ 正确。这正是多态性的体现——同样的消息（如 draw()），Circle画圆而Rectangle画矩形 |
| C. 消息的发送可以不考虑具体的接收者 | ✅ 正确。通过基类指针调用虚函数时，发送者不需要知道具体接收者类型 |
| **D. 对象必须对收到的消息做出响应** | ❌ **错误。** 对象可以响应消息，也可以选择忽略（不予理会）。响应不是强制性的 |

**📌 考点扩展：**
消息在C++中体现为**函数调用**。对象间的消息传递有三个要素：
1. **消息的发送者**（调用者）
2. **消息的接收者**（被调用的对象）
3. **消息的内容**（函数名和参数）

---

#### 1.6 面向对象方法的优点不包括：**D. 运行速度更快**

| 选项 | 分析 |
|:----:|:------|
| A. 可重用性 | ✅ 是优点。通过继承和组合实现代码重用 |
| B. 可维护性 | ✅ 是优点。封装使修改局部化，不影响其他部分 |
| C. 可扩充性 | ✅ 是优点。通过派生可以方便地扩充功能 |
| **D. 运行速度更快** | ❌ **不是优点。** OOP有少量运行时开销：动态多态需要查虚函数表（vtable），对象创建和销毁有构造/析构开销 |

**📌 考点扩展：**
面向对象的主要优点：**可重用性、可维护性、可扩充性**（三性）
主要缺点：**学习曲线较陡、运行效率略有降低、程序体积较大**

---

#### 1.7 在面向对象方法中，"对象"由哪两部分构成？**B. 属性和行为**

| 选项 | 分析 |
|:----:|:------|
| A. 函数和变量 | ❌ 这是面向过程的说法，函数和变量是分离的 |
| **B. 属性和行为** | ✅ **正确。** 对象 = 属性（数据）+ 行为（操作/函数）。属性描述对象的状态，行为描述对象能做什么 |
| C. 输入和输出 | ❌ 这是I/O概念，不是对象的构成 |
| D. 基类和派生类 | ❌ 这是继承体系中的概念 |

**📌 考点扩展：**
- 对象的三要素：**标识**（唯一身份）、**属性**（状态）、**行为**（操作）
- 公式：`对象 = 属性 + 行为 = 数据 + 操作（函数）`

#### 1.8 以下哪个是面向过程的代表性语言？**C. C**

| 选项 | 分析 |
|:----:|:------|
| A. C++ | ❌ 面向对象语言（兼容面向过程）|
| B. Java | ❌ 纯面向对象语言（一切皆对象）|
| **C. C** | ✅ **正确。** C语言以函数为核心，数据与操作分离，是面向过程的代表 |
| D. Python | ❌ 面向对象语言（也支持面向过程编程）|

**📌 考点扩展：**
语言流派对比：
- **面向过程**：C、Pascal、Fortran、BASIC
- **面向对象**：C++、Java、C#、Python、Ruby
- C++是**既支持面向过程又支持面向对象**的混合型语言（C++ = C + 类 + 其他特性）

### ● 类与对象（第1.9-1.20题）

#### 1.9 类中私有成员的特点，正确的是：**B. 只有本类成员函数和友元可以访问**

| 选项 | 分析 |
|:----:|:------|
| A. 类内类外均可访问 | ❌ 这是public成员的特点，private在类外不能访问 |
| **B. 只有本类成员函数和友元可以访问** | ✅ **正确。** private成员对外完全隐藏，只有类内部（成员函数）和特殊的"朋友"（友元）能访问 |
| C. 派生类可以访问 | ❌ 这是protected成员的特点。private成员即使在派生类中也不能直接访问 |
| D. 任何函数都可以访问 | ❌ 这是全局变量的特点，不符合封装原则 |

**📌 考点扩展：**
三种访问权限的比较：

| 权限 | 类内 | 派生类 | 类外 | 通俗理解 |
|:----:|:----:|:------:|:----:|:---------|
| **public** | ✅ | ✅ | ✅ | 对外公开，任何人都能用 |
| **protected** | ✅ | ✅ | ❌ | 对子类公开，对外隐藏 |
| **private** | ✅ | ❌ | ❌ | 只对自己公开，完全隐藏 |

---

#### 1.10 在类中，用于声明类与外部的接口的关键字是：**C. public**

| 选项 | 分析 |
|:----:|:------|
| A. private | ❌ 用于隐藏内部实现细节，类外不能访问 |
| B. protected | ❌ 用于保护成员，类外不能访问，但派生类可访问 |
| **C. public** | ✅ **正确。** public成员是类的对外接口，类内类外都可以访问 |
| D. static | ❌ static不是访问权限关键字，是存储类关键字 |

**📌 考点扩展：**
设计原则：**接口与实现分离**
- public = 接口（承诺对外提供的功能，稳定不变）
- private = 实现细节（可以随时修改，不影响使用者）
- 良好的类设计应该把**接口公开，实现隐藏**

#### 1.11 关于构造函数，下列说法正确的是：**C. 构造函数名必须与类名相同**

| 选项 | 分析 |
|:----:|:------|
| A. 构造函数可以有返回值 | ❌ 构造函数**没有返回值**，也不能指定为void类型 |
| B. 构造函数不能被重载 | ❌ 构造函数**可以重载**，一个类可以有多个参数不同的构造函数 |
| **C. 构造函数名必须与类名相同** | ✅ **正确。** 这是构造函数的最基本特点 |
| D. 构造函数必须显式定义 | ❌ 如果程序员没有定义构造函数，系统会自动生成一个**默认构造函数** |

**📌 考点扩展：**
构造函数的所有特点（必须掌握）：
1. ✅ 名称与类名相同
2. ✅ **没有返回值**（不能写void，也不能写其他类型）
3. ✅ **可以重载**（一个类有多个构造函数）
4. ✅ 可以带**默认形参值**
5. ✅ 可以是**内联函数**
6. ✅ 声明对象时**自动调用**
7. ✅ 未定义时系统自动生成**默认构造函数**（不做初始化）
8. ❌ **不能是虚函数**（因为对象还没构建完成，虚函数表还没建立）

---

#### 1.12 如果没有显式定义构造函数，系统会：**C. 自动生成一个默认构造函数，不做初始化**

| 选项 | 分析 |
|:----:|:------|
| A. 报错 | ❌ 编译器不会报错，会默认生成 |
| B. 自动生成一个带参数的构造函数 | ❌ 默认生成的是**无参**构造函数 |
| **C. 自动生成一个默认构造函数，不做初始化** | ✅ **正确。** 系统生成的默认构造函数只分配存储空间，不初始化数据成员（对于内置类型，值是随机的） |
| D. 随机生成一个构造函数 | ❌ 编译器行为是确定的，不是随机的 |

**📌 考点扩展：**
- **默认构造函数**：可以不带参数调用的构造函数（要么没有参数，要么所有参数都有默认值）
- 如果程序员定义了**任何构造函数**（哪怕是有参的），系统就不再自动生成默认构造函数
- 此时如果声明对象时不带参数，会导致**编译错误**

#### 1.13 以下关于析构函数的描述，错误的是：**C. 析构函数可以有返回值**

| 选项 | 分析 |
|:----:|:------|
| A. 析构函数没有参数 | ✅ 正确。析构函数不能有参数 |
| B. 析构函数不能重载 | ✅ 正确。一个类**有且仅有一个**析构函数 |
| **C. 析构函数可以有返回值** | ❌ **错误。** 析构函数**没有返回值**，也不能指定为void |
| D. 一个类有且仅有一个析构函数 | ✅ 正确。不能重载，所以唯一 |

**📌 考点扩展：**
构造 vs 析构 对比：

| 对比项 | 构造函数 | 析构函数 |
|--------|---------|---------|
| 名称 | 与类名相同 | `~` + 类名 |
| 返回值 | 无 | 无 |
| 参数 | 可以有 | **无参数** |
| 重载 | **可以重载** | **不能重载** |
| 调用时机 | 创建对象时自动调用 | 对象销毁时自动调用 |
| 是否可为虚函数 | **不能** | **可以**（经常需要） |

---

#### 1.14 析构函数名由什么组成？**B. `~` + 类名**

| 选项 | 分析 |
|:----:|:------|
| A. `@` + 类名 | ❌ `@`不是C++中的特殊符号 |
| **B. `~` + 类名** | ✅ **正确。** 波浪号`~`加类名，如`~Student()` |
| C. `!` + 类名 | ❌ `!`是逻辑非运算符 |
| D. `#` + 类名 | ❌ `#`是预处理器指令符号 |

**📌 考点扩展：**
口诀：**"无参无反不重载"** 这是析构函数的三个核心特点：
- **无参**：没有参数
- **无反**：没有返回值
- **不重载**：不能重载，有且仅有一个

---

#### 1.15 构造函数初始化列表的写法是：**A. `Date(int y, int m, int d) : year(y), month(m), day(d) {}`**

| 选项 | 分析 |
|:----:|:------|
| **A. `Date(int y, int m, int d) : year(y), month(m), day(d) {}`** | ✅ **正确。** 标准初始化列表语法：参数列表后跟冒号，再跟`成员名(初始值)` |
| B. `Date(int y, int m, int d) { year=y; month=m; day=d; }` | ❌ 这是**构造函数体内赋值**，不是初始化列表。对于const成员和引用成员，**必须**用初始化列表，不能体内赋值 |
| C. `Date(int y, int m, int d) = year(y), month(m), day(d)` | ❌ 语法错误，`=`不能这样用 |
| D. `Date(int y, int m, int d) :: year(y), month(m), day(d)` | ❌ `::`是作用域运算符，不是初始化列表语法 |

**📌 考点扩展：**
初始化列表 vs 函数体内赋值的区别：
- 初始化列表是**真正初始化**（调用拷贝构造函数），体内赋值是**先默认构造再赋值**
- **必须用初始化列表的场景**：
  1. **const数据成员**（常量必须在初始化时赋值）
  2. **引用数据成员**
  3. **没有默认构造函数的对象成员**
  4. **基类没有默认构造函数**（派生类必须显式调用基类构造函数）
- 初始化顺序：**按成员在类中的声明顺序**，而不是按初始化列表中的顺序

#### 1.16 当用 `delete` 释放动态对象时，系统会：**B. 先调用析构函数，再释放内存**

| 选项 | 分析 |
|:----:|:------|
| A. 释放内存，但不调用析构函数 | ❌ 如果是这样，动态对象中的资源将永远无法释放 |
| **B. 先调用析构函数，再释放内存** | ✅ **正确。** delete执行的完整流程：① 调用析构函数清理资源 → ② 释放对象占用的内存 |
| C. 只调用析构函数 | ❌ 析构函数只负责资源清理，不释放内存本体 |
| D. 什么也不做 | ❌ delete是有明确行为的 |

**📌 考点扩展：**
`new` 和 `delete` 的执行流程：
- `new`：① 分配内存 → ② 调用构造函数
- `delete`：① 调用析构函数 → ② 释放内存
- `new[]` / `delete[]`：用于对象数组，会为每个元素调用构造/析构

#### 1.17 静态局部对象（static属性）在什么时候调用析构函数？**C. main函数结束或调用exit时**

| 选项 | 分析 |
|:----:|:------|
| A. 函数调用结束时 | ❌ 静态局部对象的生命周期**不随函数调用结束而结束** |
| B. 对象被删除时 | ❌ 静态局部对象在栈上，不能用delete |
| **C. main函数结束或调用exit时** | ✅ **正确。** 静态局部对象在程序终止时才被销毁 |
| D. 永远不会调用 | ❌ 程序结束时会被销毁，只是生存期特别长 |

**📌 考点扩展：**
对象的生存期分类：
1. **局部对象**：定义在函数内，函数返回时销毁
2. **静态局部对象**（static）：定义在函数内，程序结束时销毁
3. **全局对象**：定义在所有函数外，程序结束时销毁
4. **动态对象**（new）：手动delete才销毁
5. **静态全局对象**（static + 全局）：程序结束时销毁

#### 1.18 关于对象的组合，内嵌对象构造函数的调用顺序是：**B. 按在类中声明顺序**

| 选项 | 分析 |
|:----:|:------|
| A. 按初始化列表中的顺序 | ❌ 容易混淆！初始化列表写的顺序**不影响**实际构造顺序 |
| **B. 按在类中声明顺序** | ✅ **正确。** 内嵌对象按在类中**声明的先后顺序**构造，与初始化列表顺序无关 |
| C. 按参数列表顺序 | ❌ 参数列表顺序不相关 |
| D. 随机顺序 | ❌ C++的行为是确定的 |

**📌 考点扩展：**
```cpp
class A {
    B b1;  // 先构造（声明在前）
    B b2;  // 后构造（声明在后）
public:
    A() : b2(...), b1(...) { }  // 尽管初始化列表先写b2，实际还是先构造b1
};
```
⚠️ **重要提醒**：初始化列表写的顺序如果与声明顺序不同，容易导致诡异的bug（一个成员用另一个尚未初始化的成员初始化）。**最佳实践：初始化列表顺序与声明顺序保持一致。**

#### 1.19 组合类的析构函数调用顺序与构造函数：**B. 完全相反**

| 选项 | 分析 |
|:----:|:------|
| A. 完全相同 | ❌ 析构与构造是相反的 |
| **B. 完全相反** | ✅ **正确。** 构造：先基类→再内嵌对象→后自己。析构：先自己→再内嵌对象→后基类 |
| C. 无关 | ❌ 有严格对应关系 |
| D. 随机 | ❌ C++规定得很清楚 |

**📌 考点扩展：**
重要口诀：**"构造顺序与析构顺序相反"**（后构造的先析构）
- 构造顺序：基类 → 内嵌对象（按声明顺序）→ 派生类自身
- 析构顺序：派生类自身 → 内嵌对象（逆序）→ 基类

---

#### 1.20 前向引用声明可以用于：**C. 声明该类的指针或引用**

| 选项 | 分析 |
|:----:|:------|
| A. 声明该类的对象 | ❌ 前向引用声明时，类还没完全定义，不知道大小，不能创建对象 |
| B. 在内联成员函数中使用该类的对象 | ❌ 内联函数需要知道类的完整定义，前向引用不足以支持 |
| **C. 声明该类的指针或引用** | ✅ **正确。** 指针和引用只需要知道类名（大小固定），不需要知道类的完整定义 |
| D. 定义该类的内联成员函数 | ❌ 需要类的完整定义 |

**📌 考点扩展：**
```cpp
class B;  // 前向引用声明
class A {
    B* ptr;    // ✅ 指针，可以
    B& ref;    // ✅ 引用，可以
    B func(B&); // ✅ 函数形参和返回类型，可以
    B obj;     // ❌ 对象，不行（不知道B的大小）
};
class B { /* 完整定义 */ };  // 在后面完成定义
```
前向引用常用于解决**循环依赖**问题（A类中有B类的指针，B类中有A类的指针）。

### ● 拷贝构造函数（第1.21-1.24题）

---

#### 1.21 拷贝构造函数的形参必须是：**B. 本类对象的引用**

| 选项 | 分析 |
|:----:|:------|
| A. 本类对象 | ❌ 如果形参是本类对象（值传递），调用拷贝构造函数时又要复制实参→又调用拷贝构造函数→无限递归，最终栈溢出 |
| **B. 本类对象的引用** | ✅ **正确。** 使用引用传递可以避免拷贝构造函数的无限递归。通常加const：`const Student&` |
| C. 本类对象的指针 | ❌ 虽然可以工作，但不符合拷贝构造函数的定义规范 |
| D. 任意类型 | ❌ 拷贝构造函数的形参必须是**本类对象的引用** |

**📌 考点扩展：**
```cpp
class Student {
public:
    Student(const Student& other) { /* 拷贝构造 */ }  // ✅ 标准写法
    Student(Student other) { /* 编译错误或无限递归 */ }  // ❌ 值传递会无限递归
};
```
为什么必须是引用？因为如果是值传递：
1. 用s1初始化s2 → `Student s2 = s1;`
2. 需要把s1复制给形参 → 调用拷贝构造函数
3. 拷贝构造函数的形参又是值传递 → 又需要复制...
4. **无限循环！栈溢出！**

---

#### 1.22 以下哪种情况不会调用拷贝构造函数？**D. 用对象指针初始化另一个对象指针：`Student* p2 = &s1;`**

| 选项 | 分析 |
|:----:|:------|
| A. 用对象初始化另一个对象：`Student s2 = s1;` | ✅ 会调用。这是最常见的拷贝构造场景 |
| B. 对象作为函数参数（值传递） | ✅ 会调用。实参复制给形参时调用拷贝构造函数 |
| C. 函数返回对象（值返回） | ✅ 会调用。返回的对象会拷贝给调用方。编译器可能优化为RVO（返回值优化），但语义上是拷贝构造 |
| **D. 用对象指针初始化另一个对象指针** | ❌ **不会调用。** 指针赋值只拷贝地址值，不涉及对象的创建 |

**📌 考点扩展：**
拷贝构造函数调用的**三大场景**（必考！）：
1. **初始化**：`Student s2 = s1;` 或 `Student s2(s1);`
2. **值传递**：`void func(Student s) { ... }` → `func(s1);`
3. **值返回**：`Student func() { Student s; return s; }`

不调用拷贝构造的场景：
- 引用传递、指针传递
- 赋值操作（`s1 = s2;` 调用的是赋值运算符重载，不是拷贝构造）

---

#### 1.23 当类中含有指针类型的数据成员时，通常需要：**B. 自定义拷贝构造函数实现深拷贝**

| 选项 | 分析 |
|:----:|:------|
| A. 使用默认拷贝构造函数 | ❌ 默认拷贝构造函数执行**浅拷贝**，只会复制指针的值（地址），导致两个对象指向同一块内存 |
| **B. 自定义拷贝构造函数实现深拷贝** | ✅ **正确。** 深拷贝为新对象分配独立的内存，并复制内容，避免double free |
| C. 删除拷贝构造函数 | ❌ 删除了就无法进行对象的拷贝传递，过于极端 |
| D. 将指针成员设为静态 | ❌ 静态成员是所有对象共享，不符合设计需求 |

**📌 考点扩展：**
**浅拷贝** vs **深拷贝** 对比：

| 对比项 | 浅拷贝 | 深拷贝 |
|--------|:------:|:------:|
| 操作 | 复制指针值（地址） | 复制指针所指对象本身 |
| 两个对象的数据指针 | 指向**同一块内存** | 指向**各自独立的内存** |
| 析构时 | **double free！** 同一内存释放两次 | 各自释放自己的内存 |
| 适用场景 | 无非指针成员 | 含指针类型数据成员 |

```cpp
class String {
    char* str;
public:
    // 深拷贝构造函数
    String(const String& other) {
        str = new char[strlen(other.str) + 1];  // 分配新内存
        strcpy(str, other.str);                  // 复制内容
    }
    ~String() { delete[] str; }
};
```

---

#### 1.24 浅拷贝可能导致的问题是：**B. 运行时double free（重复释放内存）**

| 选项 | 分析 |
|:----:|:------|
| A. 编译错误 | ❌ 浅拷贝在编译期不会报错，语法上是合法的 |
| **B. 运行时double free（重复释放内存）** | ✅ **正确。** 两个对象的指针成员指向同一块内存，析构时第一次释放成功，第二次释放已释放的内存→程序崩溃 |
| C. 数据成员无法初始化 | ❌ 数据成员会正常初始化（复制地址值） |
| D. 程序无法启动 | ❌ 问题在运行期析构时才会暴露，不是启动阶段 |

**📌 考点扩展：**
```cpp
// 浅拷贝的问题演示
class Array {
    int* data;
public:
    Array(int size) { data = new int[size]; }
    ~Array() { delete[] data; }  // 两个对象析构时都会执行这句
};

int main() {
    Array a1(10);
    Array a2(a1);   // 默认拷贝构造——浅拷贝
    // a1.data 和 a2.data 指向同一块内存
    return 0;       // ❌ a2析构 → delete[] data 成功
                    // ❌ a1析构 → delete[] data 再次释放 → crash!
}
```
**解决方案：** 自定义拷贝构造函数实现深拷贝，或者使用智能指针（C++11的shared_ptr）。

### ● 静态成员（第1.25-1.29题）

---

#### 1.25 静态数据成员初始化的正确方式是：**C. 在类外，对象生成之前初始化**

| 选项 | 分析 |
|:----:|:------|
| A. 在构造函数中初始化 | ❌ 静态成员属于类，不属于某个对象，构造函数中初始化会导致每次创建对象都重新赋值 |
| B. 在类定义中直接赋值 | ❌ C++规定不能在类定义中初始化静态成员（除非是const static整数类型） |
| **C. 在类外，对象生成之前初始化** | ✅ **正确。** 格式：`数据类型 类名::静态成员名 = 值;` |
| D. 在main函数中初始化 | ❌ 技术上可以赋值，但规范的初始化应在main之前完成 |

**📌 考点扩展：**
```cpp
class Student {
    static int count;  // 类内声明
};
int Student::count = 0;  // 类外初始化（通常在.cpp文件中）
```
注意：静态成员是**声明**在类内，**定义和初始化**在类外。如果不提供类外定义，链接时会报"未定义的外部符号"错误。

例外：`const static int` 类型的成员可以在类内直接初始化。

---

#### 1.26 静态成员函数的特点是：**C. 没有this指针，只能访问静态成员**

| 选项 | 分析 |
|:----:|:------|
| A. 有this指针 | ❌ 静态成员函数**没有this指针**，因为它是属于类的，不属于某个具体对象 |
| B. 可以访问非静态成员 | ❌ 没有this指针，无法确定要操作的是哪个对象的非静态成员 |
| **C. 没有this指针，只能访问静态成员** | ✅ **正确。** 正因为没有this指针，所以只能访问静态成员（静态数据和静态函数）|
| D. 只能通过对象名调用 | ❌ 静态成员函数可以**通过`类名::函数名`直接调用**，不一定要通过对象 |

**📌 考点扩展：**
```cpp
class Student {
    static int count;
    int id;
public:
    static void showCount() {
        cout << count;  // ✅ 可以访问静态成员
        cout << id;     // ❌ 不能访问非静态成员
    }
};
// 调用方式
Student::showCount();  // ✅ 通过类名调用（推荐）
Student s;
s.showCount();         // ✅ 也可以通过对象调用（但不推荐）
```

---

#### 1.27 静态数据成员属于：**B. 整个类，所有对象共享**

| 选项 | 分析 |
|:----:|:------|
| A. 每一个对象各自单独拥有 | ❌ 普通数据成员才是每个对象一份 |
| **B. 整个类，所有对象共享** | ✅ **正确。** 所有对象共享同一份静态数据，修改一个对象的静态成员，所有对象都会看到变化 |
| C. 主函数 | ❌ 不属于函数，属于类 |
| D. 全局空间 | ❌ 虽然在类外初始化，但其作用域受类名限定 |

**📌 考点扩展：**
典型应用场景：
1. **计数器**：记录当前创建了多少个对象
2. **共享配置**：所有对象共享同一个配置信息
3. **对象池**：管理所有已创建的对象
4. **单例模式**：用静态成员实现唯一实例

内存角度：静态数据成员**不在对象中**，系统单独分配存储区域。`sizeof(对象)`不包含静态成员的大小。

---

#### 1.28 静态成员函数不能访问非静态成员，原因是：**A. 静态成员函数没有this指针**

| 选项 | 分析 |
|:----:|:------|
| **A. 静态成员函数没有this指针** | ✅ **正确。** 非静态成员属于某个具体对象，静态函数不知道访问哪个对象的数据 |
| B. 静态成员函数在编译时执行 | ❌ 静态成员函数也是在运行时执行 |
| C. 静态成员函数没有参数 | ❌ 静态成员函数**可以有参数** |
| D. 静态成员函数是全局的 | ❌ 静态成员函数有类作用域限制 |

**📌 考点扩展：**
this指针的作用：每个非静态成员函数调用时，编译器自动传入当前对象的地址（this）。因此`obj.func()`可以操作obj的数据。静态函数没有这个机制，所以无法确定"当前是哪个对象"。

如果想在静态函数中访问非静态成员，唯一的方法是**通过参数传入对象**：
```cpp
static void print(const Student& s) {
    cout << s.id;  // 通过参数传入的对象访问非静态成员
}
```

---

#### 1.29 静态数据成员在内存中的存储位置：**B. 单独分配存储区域**

| 选项 | 分析 |
|:----:|:------|
| A. 和普通数据成员一样在对象中 | ❌ 静态成员不在对象中。`sizeof(类)`不包含静态成员 |
| **B. 单独分配存储区域** | ✅ **正确。** 静态数据成员存储在程序的**数据段（静态存储区）**，在程序开始执行前就分配好 |
| C. 在栈中 | ❌ 栈中存放的是局部变量和函数调用的信息 |
| D. 在寄存器中 | ❌ 寄存器是CPU内部的高速存储，不用于存储静态数据 |

**📌 考点扩展：**
C++程序内存分区：
1. **代码区**：存放程序指令
2. **全局/静态区**（数据段）：全局变量、静态变量 ⟵ 静态成员在这里
3. **栈区**：局部变量、函数参数
4. **堆区**（自由存储区）：new/malloc分配的内存

### ● 友元（第1.30-1.33题）

---

#### 1.30 下面关于友元的描述，错误的是：**B. 友元关系可以被继承**

| 选项 | 分析 |
|:----:|:------|
| A. 友元可以访问类的私有成员 | ✅ 正确。这是友元的核心功能——"破墙而入"访问私有成员 |
| **B. 友元关系可以被继承** | ❌ **错误。** 基类的友元**不是**派生类的友元。友元关系仅限于声明的那个类 |
| C. 友元关系是单向的 | ✅ 正确。A是B的友元，B不一定是A的友元（不能强求别人也把你当朋友）|
| D. 友元关系不具有传递性 | ✅ 正确。A是B的友元，B是C的友元，A不一定是C的友元（朋友的朋友不一定是朋友）|

**📌 考点扩展：**
友元关系的**三大特性**（必考！）：
1. **单向性** — 不能强求对等
2. **不具传递性** — 不能链式传递
3. **不能被继承** — 父辈的朋友不是子辈的朋友

---

#### 1.31 友元的类型包括：**A. 友元函数和友元类**

| 选项 | 分析 |
|:----:|:------|
| **A. 友元函数和友元类** | ✅ **正确。** 友元有两种形式：友元可以是**独立的函数**（友元函数），也可以是**整个类**（友元类）|
| B. 友元函数和友元对象 | ❌ "友元对象"不是标准术语 |
| C. 友元类和友元对象 | ❌ 同上 |
| D. 友元函数和友元指针 | ❌ 指针不是友元类型 |

**📌 考点扩展：**
```cpp
class A {
    friend void func();  // 友元函数：独立函数可以访问A的私有成员
    friend class B;      // 友元类：B的所有成员函数都可以访问A的私有成员
};
```

---

#### 1.32 友元的主要缺点是：**B. 破坏封装性和数据隐藏性**

| 选项 | 分析 |
|:----:|:------|
| A. 增加代码量 | ❌ 友元只是声明，不会显著增加代码量 |
| **B. 破坏封装性和数据隐藏性** | ✅ **正确。** 友元允许外部函数/类访问私有成员，破坏了OOP的封装原则。老师说：**"墙上开的小孔"** |
| C. 降低运行效率 | ❌ 恰恰相反，友元**提高**运行效率（省去了通过公有接口间接访问的开销） |
| D. 增加编译时间 | ❌ 影响不大 |

**📌 考点扩展：**
友元的**优点与缺点**对比：
- ✅ **优点**：提高运行效率、增加编程灵活性
- ❌ **缺点**：破坏封装性、降低可维护性
- **老师建议**：尽量不使用或少使用友元

---

#### 1.33 若A是B的友元类，C是A的友元类，则：**B. C不一定是B的友元类（无传递性）**

| 选项 | 分析 |
|:----:|:------|
| A. C一定是B的友元类（传递性） | ❌ 友元没有传递性 |
| **B. C不一定是B的友元类（无传递性）** | ✅ **正确。** 友元关系不具有传递性。除非B也显式声明C为友元 |
| C. C一定不是B的友元类 | ❌ 可能C同时也是B的友元（如果B声明了C为友元），但不自动是 |
| D. B一定是C的友元类（双向性） | ❌ 友元是单向的，不自动反向 |

**📌 考点扩展：**
```cpp
class B {
    friend class A;  // A是B的友元
    // A可以访问B的私有成员
};

class A {
    friend class C;  // C是A的友元
    // C可以访问A的私有成员
    // BUT: C不能自动访问B的私有成员！
};
```

### ● const与this指针（第1.34-1.38题）

---

#### 1.34 `const int *p;` 声明的是：**A. 指向常量的指针（指针指向的值不可改）**

| 选项 | 分析 |
|:----:|:------|
| **A. 指向常量的指针** | ✅ **正确。** `const`修饰`int`，说明`int`是常量，不能通过`*p`修改其值。但p本身可以指向别处 |
| B. 常量指针 | ❌ 常量指针的写法是 `int* const p`，表示p本身不能变 |
| C. 指向指针的常量 | ❌ 这是 `int** const p` |
| D. 常量指针常量 | ❌ 这是 `const int* const p` |

**📌 考点扩展：**
**const与指针的组合（必考！）：**

| 写法 | 含义 | 指针可改？ | 指向的值可改？ |
|:----|:----|:----------:|:------------:|
| `const int *p` | **指向常量的指针** | ✅ 可以 | ❌ 不行 |
| `int const *p` | 同上（const在int前后一样）| ✅ 可以 | ❌ 不行 |
| `int * const p` | **常量指针** | ❌ 不行 | ✅ 可以 |
| `const int * const p` | **指向常量的常量指针** | ❌ 不行 | ❌ 不行 |

**记忆口诀：** "const在\*前——值不可改；const在\*后——指针不可改"

---

#### 1.35 `int * const p = &x;` 声明的是：**B. 常量指针（指针本身不能改变）**

| 选项 | 分析 |
|:----:|:------|
| A. 指向常量的指针 | ❌ 这是`const int *p` |
| **B. 常量指针（指针本身不能改变）** | ✅ **正确。** `const`在`*`后面，修饰指针p本身。初始化后不能再指向其他地方 |
| C. 指向指针的指针 | ❌ 这是`int **p` |
| D. 常量引用 | ❌ 这是`const int&` |

**📌 考点扩展：**
必须初始化：常量指针必须在定义时初始化（因为之后不能改）。
```cpp
int x = 5, y = 10;
int* const p = &x;  // ✅ 初始化
p = &y;             // ❌ 编译错误：常量指针不能改指向
*p = 20;            // ✅ 可以通过指针修改指向的值
```

#### 1.36 this 指针隐含于：**B. 每一个类的非静态成员函数中**

| 选项 | 分析 |
|:----:|:------|
| A. 全局函数中 | ❌ 全局函数没有this指针 |
| **B. 每一个类的非静态成员函数中** | ✅ **正确。** this指针是C++编译器自动添加到非静态成员函数中的一个隐含参数 |
| C. 仅静态成员函数中 | ❌ **静态成员函数没有this指针！** 这是常见的易错点 |
| D. 友元函数中 | ❌ 友元函数不是类的成员函数，也没有this指针 |

**📌 考点扩展：**
```cpp
class Student {
    int id;
public:
    void set(int id) {
        this->id = id;  // this->id是成员，=右边的id是形参
    }
    static void func() {
        // this->id;   ❌ 静态函数没有this
    }
};
```
**this指针的作用：**
1. 区分同名：当形参名和成员名相同时，用`this->成员`区分
2. 返回当前对象：`return *this;`（在运算符重载和链式调用中常见）
3. 编译器隐式将当前对象的地址传递给成员函数

#### 1.37 常量引用所引用的对象：**B. 不能被修改**

| 选项 | 分析 |
|:----:|:------|
| A. 可以被修改 | ❌ 常量引用就是为了防止修改 |
| **B. 不能被修改** | ✅ **正确。** `const T&` 通过该引用不能修改原对象的值 |
| C. 可以重新赋值 | ❌ 引用一旦绑定就不能再绑定到其他对象 |
| D. 可以指向其他变量 | ❌ 引用不是指针，不能"指向其他变量" |

**📌 考点扩展：**
```cpp
const int& ref = x;  // 常量引用
ref = 10;            // ❌ 编译错误！不能通过常量引用修改
x = 10;              // ✅ 但可以直接通过x修改

// 常量引用的典型应用：函数参数
void print(const Student& s) {
    // 既避免了值传递的拷贝开销，又保证了不修改原对象
    cout << s.getName();
}
```
常量引用作为函数参数的优点：
1. **高效率**：传递引用不用拷贝对象
2. **安全性**：const保证函数内不会意外修改实参

#### 1.38 关于this指针，以下说法正确的是：**C. this指针指向当前操作的数据所属的对象**

| 选项 | 分析 |
|:----:|:------|
| A. this指针可以显式修改 | ❌ this是**常量指针**（`ClassName* const this`），不能修改其指向 |
| B. 静态成员函数也有this指针 | ❌ **静态成员函数没有this指针** |
| **C. this指针指向当前操作的数据所属的对象** | ✅ **正确。** 当调用`obj.func()`时，this就指向obj |
| D. this指针的类型是void* | ❌ this的类型是`ClassName* const`，不是void* |

**📌 考点扩展：**
```cpp
class MyClass {
    int value;
public:
    void func() {
        // 编译器自动处理，相当于：
        // void func(MyClass* const this)
        this->value = 10;  // this是MyClass* const类型
    }
};
```
this指针是**右值**，不能获取其地址，不能修改其值。

### ● 继承与派生（第1.39-1.47题）

---

#### 1.39 继承的目的和派生的目的分别是：**A. 继承为代码重用，派生为功能扩充**

| 选项 | 分析 |
|:----:|:------|
| **A. 继承为代码重用，派生为功能扩充** | ✅ **正确。** 继承是利用已有的类（基类）的代码，派生是在此基础上增加新特性 |
| B. 继承为功能扩充，派生为代码重用 | ❌ 说反了 |
| C. 都是代码重用 | ❌ 派生更侧重于扩充新功能 |
| D. 都是功能扩充 | ❌ 继承侧重于复用已有代码 |

**📌 考点扩展：**
- **继承** = 保持已有类的特性而构造新类（**重用**）
- **派生** = 在已有类的基础上新增自己的特性（**扩充**）
- 两者是一个过程的两个方面：对基类是继承，对派生类来说是派生

---

#### 1.40 在公有继承中，基类的 public 成员在派生类中变为：**C. public**

| 选项 | 分析 |
|:----:|:------|
| A. private | ❌ 那是**private继承**的效果 |
| B. protected | ❌ 那是**protected继承**的效果 |
| **C. public** | ✅ **正确。** 公有继承下，基类public成员保持不变 |
| D. 不可访问 | ❌ 只有基类的private成员在派生类中不可直接访问 |

**📌 考点扩展：**
三种继承方式下访问权限的变化总结：

| 基类中的权限 | public继承 | protected继承 | private继承 |
|:-----------:|:----------:|:------------:|:-----------:|
| **public**  | **public**  | protected    | private     |
| **protected**| protected | protected    | private     |
| **private** | 不可访问    | 不可访问      | 不可访问     |

规律：继承方式将基类的成员"降级"——**取基类权限和继承方式中更严格的那个**。

---

#### 1.41 在公有继承中，基类的 protected 成员在派生类中变为：**B. protected**

| 选项 | 分析 |
|:----:|:------|
| A. private | ❌ 那是private继承的效果 |
| **B. protected** | ✅ **正确。** 公有继承不改变protected权限 |
| C. public | ❌ 不会提升权限 |
| D. 不可访问 | ❌ protected对派生类是可见的 |

**📌 考点扩展：**
protected成员的访问特点：
- **类内**：✅ 可以访问
- **派生类**：✅ 可以访问（这是protected存在的意义）
- **类外**：❌ 不能通过对象访问

```cpp
class Base { protected: int x; };
class Derived : public Base {
    void func() { x = 10; }  // ✅ 可以访问
};
int main() {
    Derived d;
    d.x = 10;  // ❌ 类外不可访问
}
```

#### 1.42 在公有继承中，基类的 private 成员在派生类中：**D. 不可直接访问（但确实存在于对象中）**

| 选项 | 分析 |
|:----:|:------|
| A. 变为 public | ❌ private成员永远不会变成public |
| B. 变为 protected | ❌ private成员不能被派生类继承访问权限 |
| C. 保持 private | ❌ 虽然语义上"保持私有"，但派生类中确实不能直接访问它 |
| **D. 不可直接访问（但确实存在于对象中）** | ✅ **正确。** private成员在内存上确实存在（派生类对象包含基类子对象），但派生类成员函数不能直接访问它 |

**📌 考点扩展：**
```cpp
class Base {
private:
    int secret;
public:
    int getSecret() { return secret; }  // 公有接口间接访问
};

class Derived : public Base {
    void func() {
        secret = 10;    // ❌ 不能直接访问基类private成员
        getSecret();    // ✅ 可以通过公有接口间接访问
    }
};
```
派生类对象在内存中 = **基类子对象 + 派生类新增成员**。基类的private成员确实在基类子对象中，但派生类没有访问权限。

#### 1.43 派生类构造函数执行顺序是：**B. 先祖先，再客人，后自己**

| 选项 | 分析 |
|:----:|:------|
| A. 先自己，再客人，后祖先 | ❌ 这是析构的顺序 |
| **B. 先祖先，再客人，后自己** | ✅ **正确。** 经典口诀：**先祖先（基类），再客人（内嵌对象），后自己（派生类自身）** |
| C. 先祖先，再自己，后客人 | ❌ 内嵌对象在派生类自身之前构造 |
| D. 先客人，再祖先，后自己 | ❌ 基类最先构造 |

**📌 考点扩展：**
派生类构造函数的完整执行流程：
```cpp
class Derived : public Base {
    Member m1, m2;  // 内嵌对象
public:
    Derived() : Base(...), m1(...), m2(...) { /* ④ 执行派生类构造函数体 */ }
};
```
① 调用**基类构造函数**（Base）
② 按声明顺序调用**内嵌对象构造函数**（m1, m2）
③ 执行**派生类构造函数体**

口诀：**"先祖先，再客人，后自己"**

---

#### 1.44 派生类的析构函数执行顺序是：**B. 派生类 → 内嵌对象 → 基类**

| 选项 | 分析 |
|:----:|:------|
| A. 基类 → 内嵌对象 → 派生类 | ❌ 这是构造顺序 |
| **B. 派生类 → 内嵌对象 → 基类** | ✅ **正确。** 析构顺序**与构造完全相反**：先自己→再客人→后祖先 |
| C. 内嵌对象 → 基类 → 派生类 | ❌ 派生类自身最先析构 |
| D. 基类 → 派生类 → 内嵌对象 | ❌ |

**📌 考点扩展：**
构造与析构顺序对比：
```
构造顺序：Base → Member → Derived
析构顺序：Derived → Member → Base（完全相反）
```
为什么析构要相反？因为派生类的析构可能依赖于基类和内嵌对象的资源，如果基类先析构，派生类的清理就无法进行了。

#### 1.45 多继承时，基类构造函数的调用顺序是：**A. 按照派生类声明时继承的顺序（从左到右）**

| 选项 | 分析 |
|:----:|:------|
| **A. 按照派生类声明时继承的顺序（从左到右）** | ✅ **正确。** 如`class D : public A, public B`，先构造A，再构造B |
| B. 按照初始化列表中出现的顺序 | ❌ 容易混淆！和组合类一样，实际顺序由声明顺序决定 |
| C. 按照基类名的大小顺序 | ❌ 与名字无关 |
| D. 随机顺序 | ❌ C++有明确规则 |

**📌 考点扩展：**
```cpp
class A { public: A() { cout << "A"; } };
class B { public: B() { cout << "B"; } };
class C : public A, public B {  // 声明顺序：A先，B后
public:
    C() : B(), A() { }  // 初始化列表虽然先写B，但实际还是先A后B
};
// 输出：AB
```

#### 1.47 若基类声明了带参数的构造函数，则派生类必须：**B. 在初始化列表中显式调用基类构造函数**

```cpp
class Base {
```

### ● 多态性与虚函数（第1.48-1.55题）

---

#### 1.48 以下哪种多态是运行时多态？**C. 包含多态（虚函数）**

| 选项 | 分析 |
|:----:|:------|
| A. 函数重载 | ❌ 编译时确定调用的函数，是**静态多态**（静态联编）|
| B. 运算符重载 | ❌ 编译时确定，也是**静态多态** |
| **C. 包含多态（虚函数）** | ✅ **正确。** 运行时通过虚函数表动态确定调用哪个函数，是**动态多态** |
| D. 参数多态（模板） | ❌ 编译时模板实例化，是**静态多态** |

**📌 考点扩展：**
四种多态一览表：

| 类型 | 实现方式 | 绑定时机 | 一句话 |
|:----|:---------|:--------:|:-------|
| **重载多态** | 函数重载、运算符重载 | 编译时 | 同名不同参 |
| **强制多态** | 类型转换 | 编译时 | 强行变类型 |
| **包含多态** | 继承 + 虚函数 | **运行时** | 一个接口多种实现 |
| **参数多态** | 模板 | 编译时 | 类型作参数 |

#### 1.49 实现动态多态不需要以下哪个条件？**C. 使用派生类指针指向基类对象**

| 选项 | 分析 |
|:----:|:------|
| A. 基类中将函数声明为虚函数 | ✅ 是必要条件。如果没有virtual，就是静态绑定 |
| B. 公有继承（is-a关系） | ✅ 是必要条件。只有公有继承才表示is-a关系 |
| **C. 使用派生类指针指向基类对象** | ❌ **不是条件。** 正确答案应该是"使用**基类指针**指向**派生类对象**"，写反了 |
| D. 通过基类指针或引用调用虚函数 | ✅ 是必要条件。直接通过对象调用虚函数没有多态效果 |

**📌 考点扩展：**
动态多态的**三个必要条件**（必考！）：
1. **基类中将函数声明为虚函数**（virtual）
2. **公有继承**（is-a关系）
3. **基类指针/引用指向派生类对象**，并通过该指针/引用调用虚函数

**区分：**
```cpp
Base* p = new Derived();  // ✅ 基类指针指向派生类对象
p->show();                 // ✅ 运行时确定调用哪个show()

Derived* q = new Base();   // ❌ 语法错误！派生类指针不能指向基类对象
```

#### 1.50 以下哪个函数不能声明为虚函数？**C. 构造函数**

| 选项 | 分析 |
|:----:|:------|
| A. 析构函数 | ✅ **可以**是虚函数（而且经常必须声明为虚析构）|
| B. 普通成员函数 | ✅ **可以**是虚函数 |
| **C. 构造函数** | ❌ **不能。** 原因：对象还没构建完成，虚函数表还没建立，无法实现动态绑定 |
| D. 以上都可以 | ❌ 构造函数不能是虚函数 |

**📌 考点扩展：**
为什么构造函数不能是虚函数？
1. **虚函数表还没有建立**：对象在构造完成前，虚函数表（vtable）还没初始化
2. **语义不合理**：构造函数是"创建一个对象"，如果构造函数是虚函数，谁来构造它？

哪些函数不能是虚函数？**三不可**：
1. **构造函数** ❌（对象还没形成）
2. **静态成员函数** ❌（属于类，不属于对象，没有this指针）
3. **内联函数** ❌（内联是编译时展开，虚函数是运行时绑定——但编译器会忽略内联请求，将虚函数作为非内联处理）

#### 1.51 纯虚函数的定义形式是：**B. `virtual 返回值 函数名(参数表) = 0;`**

| 选项 | 分析 |
|:----:|:------|
| A. `virtual 返回值 函数名(参数表) {}` | ❌ 这是普通的**虚函数**定义（有函数体），不是纯虚函数 |
| **B. `virtual 返回值 函数名(参数表) = 0;`** | ✅ **正确。** `=0`是纯虚函数的标志，表示没有函数体 |
| C. `virtual 返回值 函数名(参数表);` | ❌ 这是**纯虚函数声明？** 不，这只是普通的成员函数声明（但没有virtual，且没有=0）|
| D. `返回值 函数名(参数表) = 0;` | ❌ 缺少virtual关键字 |

**📌 考点扩展：**
```cpp
virtual void draw() = 0;    // ✅ 纯虚函数：有=0，无函数体
virtual void draw() { }     // ✅ 普通虚函数：无=0，有函数体（可以为空）
void draw() = 0;            // ❌ 语法错误：纯虚函数必须用virtual声明
```

#### 1.52 关于抽象类，以下说法正确的是：**B. 可以声明抽象类的指针或引用**

| 选项 | 分析 |
|:----:|:------|
| A. 可以创建抽象类的对象 | ❌ **不能。** 抽象类含有纯虚函数，不能实例化 |
| **B. 可以声明抽象类的指针或引用** | ✅ **正确。** 指针和引用只是"指向"，不创建对象。可以实现多态 |
| C. 抽象类不能用作基类 | ❌ 抽象类**只能用作基类**，这是它的存在意义 |
| D. 抽象类可以实例化 | ❌ 不能实例化 |

**📌 考点扩展：**
```cpp
class Shape {  // 抽象类
public:
    virtual double area() = 0;  // 纯虚函数
};

Shape s;        // ❌ 编译错误！不能创建抽象类的对象
Shape* p;       // ✅ 可以声明指针
Shape& r = s;   // ❌ 但引用必须绑定对象，不能绑定抽象类（换个说法，不能有抽象类的引用？不对，可以通过派生类对象绑定）
Shape* arr[10]; // ✅ 可以声明指针数组

class Circle : public Shape {
    double area() override { return 3.14 * r * r; }
};
Circle c;
Shape* p = &c;  // ✅ 基类指针指向派生类对象
p->area();      // ✅ 多态调用
```

#### 1.53 虚析构函数的作用是：**B. 当基类指针指向派生类对象并用delete释放时，正确调用派生类析构函数**

| 选项 | 分析 |
|:----:|:------|
| A. 提高析构效率 | ❌ 虚函数有额外开销（查虚函数表），不会提高效率 |
| **B. 当基类指针指向派生类对象并用delete释放时，正确调用派生类析构函数** | ✅ **正确。** 这是虚析构函数的**核心作用** |
| C. 禁止析构函数被调用 | ❌ 完全相反 |
| D. 让析构函数可以被重载 | ❌ 析构函数不能重载 |

**📌 考点扩展：**
为什么要虚析构？——**防内存泄漏！**
```cpp
// 没有虚析构 → 灾难！
class Base { ~Base(); };          // 非虚
class Derived : public Base { ~Derived(); };

Base* p = new Derived();
delete p;  // ❌ 只调用了~Base()，没调用~Derived()！资源泄漏！

// 有虚析构 → 正确释放
class Base { virtual ~Base(); };  // 虚析构
Base* p = new Derived();
delete p;  // ✅ 先~Derived()，再~Base()，彻底清理
```
**规则：** 如果一个类可能作为基类被继承，并且可能通过基类指针删除派生类对象，就应该将析构函数声明为virtual。

---

#### 1.54 关于虚函数，错误的是：**D. 析构函数不能是虚函数**

| 选项 | 分析 |
|:----:|:------|
| A. 构造函数不能是虚函数 | ✅ 正确。之前解释过原因 |
| B. 静态成员函数不能是虚函数 | ✅ 正确。静态成员函数没有this指针，不能参与动态绑定 |
| C. 内联函数不能是虚函数 | ✅ 正确。 实际上，当把虚函数声明为inline时，编译器通常忽略inline请求 |
| **D. 析构函数不能是虚函数** | ❌ **错误。** 析构函数**可以**是虚函数，而且推荐在基类中声明为虚析构函数 |

**📌 考点扩展：**
**三不可**再强调一遍：
1. ❌ 构造函数不能是虚函数
2. ❌ 静态成员函数不能是虚函数
3. ❌ 内联函数不能是虚函数（编译器会忽略inline）
- ✅ 析构函数**可以**是虚函数（而且应该！）

#### 1.55 参数多态的实现方式是：**C. 模板**

| 选项 | 分析 |
|:----:|:------|
| A. 函数重载 | ❌ 这是重载多态 |
| B. 类型转换 | ❌ 这是强制多态 |
| **C. 模板** | ✅ **正确。** 函数模板和类模板都是将类型作为参数，实现代码的类型无关化 |
| D. 虚函数 | ❌ 这是包含多态（运行时多态）|

**📌 考点扩展：**
```cpp
// 函数模板 → 参数多态
template <typename T>
T max(T a, T b) { return a > b ? a : b; }

// 类模板 → 参数多态
template <typename T>
class Stack {
    T data[100];
    void push(T item);
};
```
参数多态的"多态"含义：**同一种代码模式可以处理不同类型的数据**。

### ● 运算符重载（第1.56-1.60题）

---

#### 1.56 以下运算符不能重载的是：**C. `::`**

| 选项 | 分析 |
|:----:|:------|
| A. `+` | ✅ 可以重载。是最常见的重载运算符之一（如Complex复数相加）|
| B. `-` | ✅ 可以重载 |
| **C. `::`** | ❌ **不能重载。** 作用域解析运算符，是C++语法核心，不能改变 |
| D. `<<` | ✅ 可以重载。常用于输出流（配合cout使用）|

**📌 考点扩展：**
**四个不能重载的运算符**（口诀：**点、星、域、问**）：
1. `.`（点 — 成员访问运算符）
2. `.*`（点星 — 成员指针访问运算符）
3. `::`（域 — 作用域解析运算符）
4. `?:`（问 — 三目条件运算符）
其他运算符基本都可以重载（包括`new`、`delete`、`[]`、`()`等）。

---

#### 1.57 运算符重载为成员函数时，参数个数为：**B. 原操作数个数 - 1**

| 选项 | 分析 |
|:----:|:------|
| A. 原操作数个数 | ❌ 这是**友元函数**形式的参数个数 |
| **B. 原操作数个数 - 1** | ✅ **正确。** 因为第一个操作数通过隐式的this指针传递（当前对象）|
| C. 原操作数个数 + 1 | ❌ 不会多出参数 |
| D. 0 | ❌ 至少需要-1后的参数个数 |

**📌 考点扩展：**
```cpp
// 成员函数重载二元运算符+
Complex operator+(const Complex& c);  // 1个参数（二元运算符-1=1）
// c1 + c2 → c1.operator+(c2)  // c1是this指向的对象

// 成员函数重载一元运算符-
Complex operator-();  // 0个参数（一元运算符-1=0）
// -c1 → c1.operator-()
```

---

#### 1.58 运算符重载为友元函数时，参数个数为：**A. 原操作数个数**

| 选项 | 分析 |
|:----:|:------|
| **A. 原操作数个数** | ✅ **正确。** 友元函数没有this指针，所有操作数都通过参数传递 |
| B. 原操作数个数 - 1 | ❌ 这是成员函数形式 |
| C. 原操作数个数 + 1 | ❌ |
| D. 0 | ❌ |

**📌 考点扩展：**
```cpp
// 友元函数重载二元运算符+
friend Complex operator+(const Complex& a, const Complex& b);  // 2个参数
// c1 + c2 → operator+(c1, c2)

// 友元函数重载一元运算符-
friend Complex operator-(const Complex& a);  // 1个参数
// -c1 → operator-(c1)
```
友元函数的优势：左操作数可以是其他类型（如 `3 + c`），而成员函数要求左操作数必须是本类对象。

---

#### 1.59 以下运算符重载规则，错误的是：**C. 可以发明新的运算符**

| 选项 | 分析 |
|:----:|:------|
| A. 不能改变运算符的优先级 | ✅ 正确。重载后的优先级与原来相同 |
| B. 不能改变操作数个数 | ✅ 正确。一元还是一元，二元还是二元 |
| **C. 可以发明新的运算符** | ❌ **错误。** 只能重载C++已有的运算符，不能自定义新的符号 |
| D. 至少一个操作数是自定义类型 | ✅ 正确。不能改变内置类型的运算规则 |

**📌 考点扩展：**
运算符重载的**核心规则**（必考！）：
1. ❌ 不能重载：`.` `.*` `::` `?:`
2. ❌ 不能发明新运算符
3. ✅ 不改变优先级和结合性
4. ✅ 不改变操作数个数
5. ✅ 至少一个操作数是自定义类型
6. ✅ 不能改变内置类型的运算
7. ✅ 实质是函数重载：`operator 运算符(参数)`

---

#### 1.60 如果用友元函数重载 `operator+` 实现 `c1 + c2`，函数原型是：**B. `friend Complex operator+(const Complex &a, const Complex &b);`**

| 选项 | 分析 |
|:----:|:------|
| A. `Complex operator+(Complex &c);` | ❌ 成员函数形式，且参数应该加const |
| **B. `friend Complex operator+(const Complex &a, const Complex &b);`** | ✅ **正确。** 友元函数形式，两个常量引用参数 + friend声明 |
| C. `Complex operator+(Complex &c1, Complex &c2);` | ❌ 虽然是两个参数，但没有friend关键字，如果在类外定义无法访问私有成员 |
| D. `friend Complex operator+();` | ❌ 无参数，无法实现二元运算 |

**📌 考点扩展：**
```cpp
// 成员函数
class Complex {
    Complex operator+(const Complex& c) const;
    // c1.operator+(c2)
};

// 友元函数
class Complex {
    friend Complex operator+(const Complex& a, const Complex& b);
    // operator+(c1, c2)
};
```
**友元函数 vs 成员函数的选择原则：**
- 如果左操作数必须是本类对象 → 用**成员函数**
- 如果左操作数可以是其他类型 → 用**友元函数**（如 `3 + c`）
- 需要修改左操作数 → 用**成员函数**
- 推荐：单目运算符用成员函数，双目运算符用友元函数

#### 1.63 在UML中，表示继承关系的术语是：**C. 泛化关系**

#### 1.64 以下哪种关系适合用UML中的聚合关系表示？**B. 汽车-轮胎（has-a）**

| A. 学生-研究生（is-a） | ❌ is-a关系用**泛化（继承）**表示，不是聚合 |
| **B. 汽车-轮胎（has-a）** | ✅ **正确。** 汽车"有"轮胎，是整体-部分关系，用**聚合（空心菱形）**表示 |
| C. 形状-圆形 | ❌ 圆形"是"一种形状，用**泛化**表示 |
| D. 动物-狗 | ❌ 狗"是"一种动物，用**泛化**表示 |

**📌 考点扩展：**

| 关系 | 语义 | UML图示 | 示例 |
|:----|:-----|:-------:|:----|
| **泛化** | is-a（是一种） | 空心三角 | 研究生is-a学生 |
| **聚合** | has-a（有一个）松散 | **空心菱形** | 班级has-a学生（班级没了学生还存在）|

| **B. 继承** | ✅ **正确。** 泛化（Generalization）就是C++中的public继承 |
| C. 多态 | ❌ 多态是运行时的行为，泛化是编译时的类型关系 |
| D. 模板 | ❌ 模板是参数多态，与泛化无关 |

**泛化关系在C++中的体现：**

UML中的其他关系与C++对应：
- **依赖** → 函数形参、局部变量
- **关联** → 成员指针/引用
- **泛化** → **公有继承**
- **实现** → 纯虚函数/接口

### ● 引用（第1.66-1.70题）

#### 1.66 关于引用，以下说法正确的是：**B. 引用是变量的别名，对引用的操作等价于对被引用变量的操作**

| 选项 | 分析 |
|:----:|:------|
| A. 引用是一个独立的变量，有自己独立的内存空间 | ❌ 引用本身不独立占用内存（底层通过指针实现，但逻辑上是别名）|
| **B. 引用是变量的别名，对引用的操作等价于对被引用变量的操作** | ✅ **正确。** 这是引用的核心定义。如`int& ref = a; ref = 10;`等价于`a = 10;`|
| C. 引用可以不初始化 | ❌ **引用必须初始化**，没有"空引用"的概念 |
| D. 引用可以重新绑定到另一个变量 | ❌ 引用一旦绑定，**不能改变指向**。`ref = b;`是将b的值赋给a，不是让ref改指向b |

**📌 考点扩展：**
引用 = 别名。关键规则：**必须初始化，不能重绑定，没有空值**。

---

#### 1.67 以下函数参数传递方式中，可以直接修改实参且语法最简洁的是：**C. 引用传递 `void func(int& a)`**

| 选项 | 分析 |
|:----:|:------|
| A. 值传递 `void func(int a)` | ❌ 值传递拷贝实参，**不能修改**原实参 |
| B. 指针传递 `void func(int* a)` | ✅ 可以修改实参，但语法繁琐（需要`*`解引用、调用时需`&`取地址）|
| **C. 引用传递 `void func(int& a)`** | ✅ **最简洁。** 形参声明时加`&`，调用时和普通变量一样使用，修改直接反映到实参 |
| D. 以上都不行 | ❌ C可以 |

**📌 考点扩展：**
```cpp
void swap1(int a, int b) { int t=a; a=b; b=t; }        // ❌ 不修改实参
void swap2(int* a, int* b) { int t=*a; *a=*b; *b=t; }  // ✅ 但需 & 和 *
void swap3(int& a, int& b) { int t=a; a=b; b=t; }       // ✅ 最简洁！推荐
```

#### 1.68 关于引用与指针的区别，错误的是：**C. 引用有空值，指针有nullptr**

| 选项 | 分析 |
|:----:|:------|
| A. 引用必须初始化，指针可以不初始化 | ✅ 正确。引用声明时必须绑定变量，指针可以先声明后赋值 |
| B. 引用绑定后不能改变，指针可以改变指向 | ✅ 正确。引用一生只跟一个变量绑定 |
| **C. 引用有空值，指针有nullptr** | ❌ **错误。引用不存在空值！** 引用必须绑定有效变量，没有"空引用"的概念。指针可以有nullptr |
| D. 引用不需要解引用操作，指针需要`*`解引用 | ✅ 正确。引用直接用变量名，指针需要`*`访问所指对象 |

**📌 考点扩展：**
| 区别点 | 引用 | 指针 |
|--------|:----:|:----:|
| 初始化 | 必须 | 可以不初始化（危险）|
| 重绑定 | ❌ 不能 | ✅ 能 |
| 空值 | ❌ 不存在 | ✅ 有nullptr |
| 解引用 | 自动 | `*`手动 |
| 多级 | ❌ 无 | ✅ `**p` |
| 算术运算 | ❌ 无 | ✅ `++、--、+n` |

#### 1.69 `const int& ref = x;` 以下操作正确的是：**C. `cout << ref;`**

| 选项 | 分析 |
|:----:|:------|
| A. `ref = 10;` | ❌ 常量引用不能通过引用修改被引用的对象 |
| B. `cin >> ref;` | ❌ `>>`是输入，会修改ref的值，但const禁止修改 |
| **C. `cout << ref;`** | ✅ **正确。** 读取const引用是允许的，不会修改值 |
| D. `ref++;` | ❌ 自增操作会修改ref的值，const禁止 |

**📌 考点扩展：**
`const T&` 作为函数参数的**两大优势**：
1. **避免拷贝**：传递引用不需要复制整个对象（大对象时节省巨大开销）
2. **保护数据**：const保证函数内不会意外修改原数据

#### 1.70 若函数返回引用，以下说法错误的是：**D. 返回引用一定比返回值更安全**

| 选项 | 分析 |
|:----:|:------|
| A. 返回引用可以避免拷贝，提高效率 | ✅ 正确。引用返回不产生临时对象拷贝 |
| B. 不能返回局部变量的引用 | ✅ 正确。局部变量在函数结束时就销毁了，返回其引用导致悬空引用 |
| C. 返回引用可以作为左值 | ✅ 正确。如 `getMax(a,b) = 100;` 可以直接赋值 |
| **D. 返回引用一定比返回值更安全** | ❌ **错误。** 如果返回局部变量的引用，比返回值更危险（返回值是拷贝，安全；返回引用是别名，局部变量销毁后引用悬空） |

**📌 考点扩展：**
```cpp
int& bad() {
    int x = 10;
    return x;  // ❌ 危险！x离开函数被销毁，返回的引用悬空
}
int good() {
    int x = 10;
    return x;  // ✅ 安全，返回的是x的拷贝
}
```
**规则：** 返回引用时，必须确保被引用的对象在函数返回后仍然存在（如全局变量、静态变量、传入的引用参数、对象成员等）。

| 2.1 | **属性，行为**（或：数据，操作） | 对象的两大构成：属性描述状态（有什么），行为描述能力（能做什么）。公式：`对象 = 数据 + 操作` |

| 2.3 | **类名，返回值** | 构造函数名与类名完全一致，没有任何返回值类型（连void也不行）。这是C++语法的硬性规定 |
| 2.4 | **~，重载** | 析构函数 = `~类名()`。特点：**无参、无反、不能重载**（口诀）。一个类有且仅有一个析构函数 |
| 2.5 | **成员函数，当前操作的数据所属的对象** | this指针是编译器自动添加的隐含参数，类型为`ClassName* const`。静态成员函数**没有**this指针。常用于：区分形参与成员名、返回`*this`实现链式调用 |
| 2.6 | **类外，前** | 静态数据成员的初始化格式：`类型 类名::静态成员 = 值;`。必须在main函数执行前、任何对象创建前完成初始化 |
| 2.7 | **this，静态** | 静态成员函数没有this指针→无法确定操作哪个对象的非静态数据→只能访问静态成员。调用方式：`类名::静态函数名()` |
| 2.8 | **单，传递，继承** | 友元三大特性：① **单向**（A是B的友元，B不一定也是A的友元）② **不传递**（A是B的友元，B是C的友元，A不一定是C的友元）③ **不继承**（基类的友元不是派生类的友元） |
| 2.9 | **代码重用，功能扩充** | 继承 ≈ 复用父类代码（不修改直接使用），派生 ≈ 扩充新功能（在父类基础上增加新特性）。这是一个硬币的两面 |
| 2.10 | **祖先（基类），客人（内嵌对象），自己（派生类）** | 口诀："先祖先，再客人，后自己"。构造顺序：Base → Member（按声明顺序）→ Derived。析构顺序完全相反 |
| 2.11 | **完全不同** | 多态的定义：同样的消息（如draw()）被不同类型对象（Circle / Rectangle）接收时，产生完全不同的行为（画圆 / 画矩形）。"一个接口，多种状态" |
| 2.12 | **纯虚函数，不能，能** | 抽象类的特征：含至少一个纯虚函数（`=0`）。**不能**创建抽象类的对象，但**能**声明指针和引用—这是实现多态的基础 |
| 2.13 | **.（点），.\*（点星），::（域），?:（问号）** | 口诀："**点、星、域、问**"——四个不能重载的运算符。`.`成员访问、`.*`成员指针访问、`::`作用域解析、`?:`三目条件 |
| 2.14 | **标准错误，无，日志，有** | cout=有缓冲的常规输出；cerr=**无缓冲**的错误输出（立即显示，程序崩溃也不丢失）；clog=有缓冲的日志输出 |
| 2.15 | **提取，插入** | `>>`是提取运算符（从流中读取数据），`<<`是插入运算符（向流中写入数据）。重载这两个运算符可以实现自定义类型的文件I/O |
| 2.16 | **统一建模语言（Unified Modeling Language），可视化** | UML是软件工程中用于**分析、设计、文档化**的可视化建模语言。不是编程语言，不能直接编译执行 |
| 2.17 | **泛化（generalization），关系** | 泛化（空心三角箭头）对应C++的**继承**机制（is-a关系）。UML三大基本组成部分：**事物**（模型元素）、**关系**（事物间的联系）、**图**（事物的集合） |
| 2.18 | **别名，初始化** | 引用是一个已存在变量的**别名**。声明引用时**必须初始化**（int& ref = x;），没有空引用。一旦绑定，不能改变 |
| 2.19 | **内存空间（地址），影响** | 引用传递时形参是实参的别名，两者指向同一块内存。对形参的修改**直接改变实参**的值。与指针传递效果相同，但语法更简洁 |
| 2.20 | **const，不能** | 常量引用格式：`const 类型& 引用名`。通过常量引用**不能修改**被引用的对象。常用于函数参数：`void func(const string& s)` |

### 3.1 面向对象的四大基本特征

| 特征 | 含义 | 一句话 |
|:----|:------|:-------|
| **① 抽象性** | 对同一类对象的共同属性和行为进行概括，抽出共性形成类。**抽象是面向对象的基础** | 找共同点 |
| **② 封装性** | 将数据及对数据的操作方法封装在一起，对外只暴露简单接口，隐藏内部实现细节。用访问权限（public/protected/private）控制可见性 | 装起来、藏起来 |
| **③ 继承性** | 保持已有类的特性而构造新类的过程。子类自动获得父类的属性和行为，实现**代码重用** | is-a，复用 |
| **④ 多态性** | 发出同样的消息被不同类型的对象接收时可能导致完全不同的行为。"一个接口，多种状态" | 同名不同行为 |

### 3.2 构造函数的特点

构造函数是一种特殊的成员函数，用于在声明对象时对数据成员进行初始化。

特点（至少列出5条）：
1. **名称与类名相同**——这是最基本的特点
2. **没有返回值类型**——也不能指定为void类型
3. **可以有参数，可以重载**——一个类可以有多个构造函数（无参、有参、拷贝构造）
4. **可带默认形参值**——如`Student(string n = "unknown")`
5. **可以是内联函数**——短小的构造函数可以内联
6. **未定义时系统自动生成默认构造函数**——不做初始化，但分配存储空间
7. **声明对象时自动调用**——不需要手动调用
8. **构造函数不能是虚函数**——因为对象还没构建完成，虚函数表还没建立

**评分标准：** 答出定义给1分，每列出1个特点给1分，最多4分，满分5分。

### 3.3 拷贝构造函数

**定义：** 拷贝构造函数是一种特殊的构造函数，用**已有对象初始化新对象**。形参是本类对象的**引用**（const引用更安全）。

**调用场景（三种）：**
1. **用对象初始化另一个对象**：`Student s2 = s1;` 或 `Student s2(s1);`
2. **对象作为函数参数（值传递）**：`void func(Student s) { ... }` → `func(s1);`
3. **函数返回对象（值返回）**：`Student func() { Student s; return s; }`

**为什么需要自定义深拷贝？**
- 默认拷贝构造函数执行**浅拷贝**——只复制数据成员的值
- 如果类中有**指针类型的数据成员**，浅拷贝只是复制地址值，导致两个对象的指针指向同一块内存
- 析构时第一个对象释放了内存，第二个对象再次释放同一块内存 → **double free → 程序崩溃**
- 自定义深拷贝：为新对象**分配独立内存**，复制内容，各自管理自己的内存

```cpp
class String {
    char* str;
public:
    String(const String& other) {
        str = new char[strlen(other.str) + 1];  // 分配新内存
        strcpy(str, other.str);                  // 复制内容
    }
};
```

**评分标准：** 答出定义给1分，列出三种场景各给1分，说明深拷贝原因给2分，满分5分。

### 3.4 静态成员

**静态数据成员的特点：**
- 属于**整个类**，被所有对象**共享**，不属于某个特定对象
- 在**类外初始化**，且在对象生成之前完成（格式：`类型 类名::成员 = 值;`）
- 系统为静态数据成员**单独分配存储区域**（数据段），不在对象中
- `sizeof(对象)` 不包含静态成员的大小

**静态成员函数的特点：**
- 没有 **this 指针**（所以不能访问非静态成员）
- 只能访问本类中的**静态成员**（静态数据成员和静态成员函数）
- 可以通过 **`类名::函数名`** 直接调用（不依赖于对象）
- 也可以通过 **`对象名.函数名`** 调用（但不推荐）
- 常用于操作静态数据成员（如计数器）

```cpp
class Student {
    static int count;    // 静态数据成员声明
public:
    Student() { count++; }
    static int getCount() { return count; }  // 静态成员函数
};
int Student::count = 0;  // 类外初始化

cout << Student::getCount();  // 通过类名调用
```

**评分标准：** 答出静态数据成员特点给2分，静态成员函数特点给2分，表达完整给1分，满分5分。

### 3.5 友元

**定义：** 友元是C++提供的一种破坏数据封装和数据隐藏的机制。通过将一个模块声明为另一个模块的友元，该模块能够访问另一个模块中的私有成员。老师比喻：**"墙上开的小孔"**。

**优点：** 提高程序的运行效率（省去公有接口的间接调用开销），增加编程灵活性。
**缺点：** 破坏了类的**封装性**和数据的**隐藏性**，降低了程序的可维护性。**建议：尽量不使用或少使用。**

**友元关系的三个特性（必背！）：**
1. **单向性** — A是B的友元，B不一定是A的友元（朋友不一定互认）
2. **不具传递性** — A是B的友元，B是C的友元，A不一定是C的友元（朋友的朋友≠朋友）
3. **不能被继承** — 基类的友元不是派生类的友元（父辈的朋友≠子辈的朋友）

```cpp
class B {
    friend class A;       // A是B的友元类
    friend void func();   // func是B的友元函数
    // 现在A和func都可以访问B的私有成员了
};
```

**评分标准：** 答出定义给1分，答出优缺点各给1分，答出三个特性给2分，满分5分。

### 3.6 三种访问权限

| 访问权限 | 类内访问 | 派生类访问 | 类外访问 | 通俗理解 | 场景举例 |
|:--------:|:--------:|:----------:|:--------:|:---------|:---------|
| **public** | ✅ | ✅ | ✅ | **对外公开**，任何人都能用 | 类的接口函数如 `getArea()` |
| **protected** | ✅ | ✅ | ❌ | **对子公开**，对外隐藏 | 派生类需要继承的成员如 `radius` |
| **private** | ✅ | ❌ | ❌ | **只对己公开**，完全隐藏 | 内部实现细节如 `secretData` |

**一句话总结：** public 对外公开，protected 对子类公开，private 只对自己公开。

**设计原则：** 好的类设计应该——接口（public）稳定，实现（private）可以随时修改。

**评分标准：** 能正确说明三种权限的访问范围给3分，能举例说明给2分，满分5分。

### 3.7 动态多态的三个条件

实现动态多态（运行时多态）需要：

1. **基类中将函数声明为虚函数（virtual）** — 告诉编译器这个函数需要动态绑定，不要编译时决定
2. **类之间存在 is-a 关系（公有继承）** — 只有public继承才表示is-a关系。私有继承和保护继承不能实现多态
3. **使用基类指针或引用指向派生类对象，并通过该指针或引用调用虚函数** — 程序运行时根据**实际对象类型**决定调用哪个函数版本（通过**虚函数表vtable**实现）

```cpp
class Shape {
public:
    virtual void draw() { cout << "Shape"; }  // 条件1: virtual
};
class Circle : public Shape {                 // 条件2: public继承
    void draw() override { cout << "Circle"; }
};
int main() {
    Shape* p = new Circle();   // 条件3: 基类指针指向派生类对象
    p->draw();                 // 运行时决定 → 输出 "Circle"
}
```

**核心：** 编译看指针类型（Shape*），运行看对象类型（Circle）。

**评分标准：** 答出每个条件给1.5分，结合虚函数说明给0.5分，满分5分。

### 3.8 纯虚函数与抽象类

**纯虚函数定义：** `virtual 返回值类型 函数名(参数表) = 0;` 只给出函数声明，没有给出函数实现，由派生类负责实现。

**抽象类：** 包含**至少一个纯虚函数**的类称为抽象类。

**特点：**
- 抽象类**只能用作基类**（不能单独实例化）
- **不能创建抽象类的对象**，但**可以声明抽象类的指针和引用**——这是实现多态的关键
- 派生类必须**实现所有纯虚函数**才能成为非抽象类（才能实例化）
- 如果派生类没有实现全部纯虚函数，它仍然是抽象类

**作用：**
- 为类族提供**统一的公共接口**
- **强制**派生类实现特定功能（不实现就不能实例化）
- 实现**面向接口编程**的设计思想

```cpp
class Shape {                        // 抽象类
public:
    virtual double area() const = 0; // 纯虚函数
    virtual void draw() const = 0;   // 纯虚函数
};
class Circle : public Shape {
public:
    double area() const override { return 3.14 * r * r; }
    void draw() const override { cout << "○"; }
};

Shape s;        // ❌ 不能创建抽象类对象
Shape* p;       // ✅ 可以声明指针
p = new Circle; // ✅ 指向派生类对象
```

**评分标准：** 答出纯虚函数定义给1分，抽象类定义给1分，特点给2分，作用给1分，满分5分。

### 3.9 静态联编 vs 动态联编

| 对比项 | 静态联编（静态绑定） | 动态联编（动态绑定） |
|:------|:------------------:|:------------------:|
| **定义** | 编译阶段就确定要调用的函数 | 程序运行时才确定调用的函数 |
| **实现方式** | 函数重载、运算符重载 | 继承 + 虚函数 |
| **绑定时机** | **编译时** | **运行时** |
| **效率** | 高（无额外运行时开销） | 有少量开销（查虚函数表 vtable）|
| **灵活性** | 差（固定调用哪个函数） | 好（根据实际对象类型动态决定）|
| **典型应用** | 普通函数调用、重载函数 | 基类指针调用虚函数 |

**核心区别一句话：** 静态联编**编译时决定**，动态联编**运行时决定**。

```cpp
// 静态联编：编译时确定调用Base::show()
void func(Base b) { b.show(); }

// 动态联编：运行时根据p指向的对象类型决定
void func(Base* p) { p->show(); }  // 需要show()是virtual
```

**评分标准：** 正确解释两种联编各给1.5分，指出核心区别给2分，满分5分。

### 3.10 运算符重载规则

**规则（至少列出6条）：**
1. **不能重载的运算符**：`.` `.*` `::` `?:`（口诀：**点、星、域、问**）
2. **不能发明新的运算符**，只能重载C++已有的
3. **不改变原运算符的优先级**
4. **不改变原运算符的结合性**
5. **不能改变操作数个数**（一元还是一元，二元还是二元）
6. **至少有一个操作数是自定义类型**（防止改变内置类型的运算）
7. 运算符重载实质上是**函数的重载**，使用 `operator` 关键字
8. 重载不能改变运算符作用于内置类型对象时的含义

**成员函数 vs 友元函数的区别：**

| 对比项 | 成员函数 | 友元函数 |
|--------|:-------:|:--------:|
| **参数个数** | 原操作数个数 - 1（this隐式充当左操作数） | 原操作数个数 |
| **左操作数限制** | 必须是本类对象 | 可以是其他类型（如`3 + c`）|
| **示例：c1 + c2** | `c1.operator+(c2)` | `operator+(c1, c2)` |
| **是否需friend声明** | 不需要 | 需要在类内声明为friend |
| **支持 `3 + c`** | ❌ 不行 | ✅ 可以（因为3可以作为第一个参数传入）|

**选择原则：** 如果左操作数可以是其他类型，或者需要实现交换律（如`3 + c`和`c + 3`），用友元函数。

**评分标准：** 规则每列出1条给0.5分，满分3分；两种形式的区别答对给2分，满分5分。

### 3.11 UML

**什么是UML？**
UML（Unified Modeling Language，**统一建模语言**）是一种**可视化**的面向对象建模语言。它不是编程语言，不能直接编译执行，而是用于软件系统的**分析、设计和文档化**的标准图形化表示方法。

**三种基本组成部分：**

| 组成部分 | 说明 | 举例 |
|:---------|:-----|:------|
| **① 事物**（Things） | 模型中的基本元素 | 类、接口、用例、组件、节点等 |
| **② 关系**（Relationships） | 事物之间的联系 | 依赖、关联、泛化、实现 |
| **③ 图**（Diagrams） | 事物的集合，从不同角度描述系统 | 类图、用例图、时序图等 |

**UML中的四种关系：**

| 关系 | 图示 | 说明 | C++对应 |
|:----|:----:|:------|:--------|
| **依赖**（Dependency） | 虚线箭头 ➡ | 一个类使用另一个类（函数参数、局部变量） | 形参、局部对象 |
| **关联**（Association） | 实线箭头 ➡ | 类之间有结构化的长期连接 | 成员指针/引用 |
| **泛化**（Generalization） | 空心三角实线 △—▶ | 一般-特殊关系（is-a） | **继承** ✅ |
| **实现**（Realization） | 空心三角虚线 △···▶ | 接口与实现的关系 | 抽象类派生 |

其中，**泛化关系**对应C++中的**继承**机制（public继承的is-a关系）。

**评分标准：** 正确解释UML定义给1分，说出三种组成部分给2分，说出四种关系及对应C++机制给2分，满分5分。

### 3.12 三种参数传递方式对比

| 对比项 | 值传递 | 指针传递 | 引用传递 |
|:------|:------:|:--------:|:--------:|
| **是否修改实参** | ❌ 不修改 | ✅ 可以修改 | ✅ 可以修改 |
| **语法简洁度** | ✅ 最简洁 | ❌ 需`*`和`&` | ✅ 简洁 |
| **安全风险** | ✅ 无 | ❌ 可能空指针 | ✅ 无空值 |
| **大对象拷贝** | ❌ 有拷贝开销 | ✅ 无拷贝 | ✅ 无拷贝 |
| **调用方式** | `func(x)` | `func(&x)` | `func(x)` |

**const引用作参数的两大优势：**
1. **高效率** — 引用传递避免了大对象的拷贝开销（传引用只需4/8字节，不管对象多大）
2. **安全性** — const保证函数内部不会意外修改传入的实参

```cpp
void print(const string& s) {  // ✅ 推荐：不拷贝 + 不改原数据
    cout << s;
}
void print(string s) {  // ❌ 传值：拷贝整个string，开销大
    cout << s;
}
```

**评分标准：** 正确说明三种传递方式各给1分，说明const引用优势给2分，满分5分。

**逐行解析：**
1. `Base* p; Base b; Derived d;` — 声明基类指针、基类对象、派生类对象
2. `p = &b; p->show();` → p指向Base对象，调用Base::show() → 输出 **Base**
3. `p = &d; p->show();` → p指向Derived对象，由于show()是**虚函数**（virtual），**动态绑定** → 调用Derived::show() → 输出 **Derived**

**知识点：** 虚函数与动态多态——通过基类指针调用虚函数时，实际调用的是**指针所指向的对象所属类**的函数版本。如果show()不是虚函数，则两次都输出"Base"（静态绑定，按指针类型决定）。

---

### 4.2 运行结果

```
ID: 1, Total: 3
ID: 2, Total: 3
ID: 3, Total: 3
```

**逐行解析：**
1. `static int total = 0;` — 静态数据成员，**所有对象共享**，初始值为0
2. `Student s1;` → 构造函数调用：total++（total=1），id=total=1
3. `Student s2;` → 构造函数调用：total++（total=2），id=total=2
4. `Student s3;` → 构造函数调用：total++（total=**3**），id=total=3
5. `s1.show()` → id=1, total=3 → 输出 **ID: 1, Total: 3**
6. `s2.show()` → id=2, total=3 → 输出 **ID: 2, Total: 3**
7. `s3.show()` → id=3, total=3 → 输出 **ID: 3, Total: 3**

**关键理解：** total是**静态成员**，所有对象共享同一份。三个对象创建后total都变成3。而id是**普通成员**，每个对象在创建时从total获取自己的id值，所以各不同。

### 4.3 运行结果

```
Base constructor
Derived constructor
Derived destructor
Base destructor
```

**逐行解析：**
1. `Derived d;` 构造过程：
   - ① 先调用**基类构造函数** → 输出 **Base constructor**
   - ② 再调用**派生类构造函数** → 输出 **Derived constructor**
2. `return 0;` 析构过程（与构造相反）：
   - ① 先调用**派生类析构函数** → 输出 **Derived destructor**
   - ② 再调用**基类析构函数** → 输出 **Base destructor**

**知识点：** 派生类构造/析构顺序——口诀"先祖先，再客人，后自己"，析构顺序完全相反。

---

### 4.4 运行结果

```
Base constructor
Derived constructor
Base destructor
```

**逐行解析：**
1. `new Derived()` → 先调Base() → 输出 **Base constructor**，再调Derived() → 输出 **Derived constructor**
2. `delete p;`（p是Base*类型）：
   - 析构函数**不是虚函数** → **静态绑定** → 根据指针类型（Base*）决定调用Base的析构函数
   - 输出 **Base destructor**
   - **派生类的析构函数没有被调用！** 派生类中动态分配的资源无法释放 → **内存泄漏！**

**与4.8对比：** 本题析构函数没有virtual关键字，导致delete基类指针时只调用了基类析构，派生类部分未清理。4.8加了virtual后就能正确调用派生类析构了。

### 4.5 运行结果

```
Shape
Circle
Rectangle
```

**逐行解析：**
1. `Shape* shapes[3];` — 声明基类指针数组
2. `shapes[0] = &s;` — 指向Shape对象
3. `shapes[1] = &c;` — 指向Circle对象
4. `shapes[2] = &r;` — 指向Rectangle对象
5. 循环三次调用 `shapes[i]->draw()`：
   - 由于draw()是**虚函数**，**动态绑定**到实际对象的类型
   - shapes[0] → Shape对象 → 输出 **Shape**
   - shapes[1] → Circle对象 → 输出 **Circle**
   - shapes[2] → Rectangle对象 → 输出 **Rectangle**

**知识点：** 这是多态数组的经典应用——通过基类指针数组管理多种派生类对象，统一调用接口。

### 4.6 运行结果

```
Base a=10
Derived a=20, b=30
```

**逐行解析：**
1. `Base base(10);` — 调用Base构造函数，a=10
2. `Derived derived(20, 30);` — 调用Derived构造函数：先调Base(20)使基类部分a=20，再设置b=30
3. `base.show();` — 调用Base::show() → 输出 **Base a=10**
4. `derived.show();` — 调用Derived::show() → 输出 **Derived a=20, b=30**

**注意：** show()函数**没有virtual关键字**，所以是**静态绑定**。通过对象直接调用成员函数时，始终调用该对象类型对应的版本。无论show()是不是虚函数，`derived.show()`都会调用Derived的版本——多态只体现在通过指针/引用调用虚函数时。

派生类show()中可以访问`a`，因为a在Base中是protected成员，对派生类公开。

### 4.7 运行结果

```
2.5+3i
1.5+2i
4+5i
```

**逐行解析：**
1. `Complex c1(2.5, 3.0);` — 创建复数 c1 = 2.5 + 3i
2. `Complex c2(1.5, 2.0);` — 创建复数 c2 = 1.5 + 2i
3. `Complex c3 = c1 + c2;` — 调用成员函数 operator+：
   - `c1.operator+(c2)` → this指向c1
   - real = 2.5 + 1.5 = 4.0，imag = 3.0 + 2.0 = 5.0
   - 返回 Complex(4.0, 5.0)，赋值给 c3
4. `c1.display()` → 输出 **2.5+3i**
5. `c2.display()` → 输出 **1.5+2i**
6. `c3.display()` → 输出 **4+5i**

**知识点：** 运算符重载为成员函数——左操作数c1通过this传入，右操作数c2作为参数传入。

### 4.8 运行结果

```
Base()
Derived()
~Derived()
~Base()
```

**逐行解析：**
1. `new Derived()` 构造：
   - ① 先调 Base() → 输出 **Base()**
   - ② 再调 Derived() → 输出 **Derived()**
2. `delete p;`（p是Base*类型）析构：
   - 基类析构函数是**虚函数**（`virtual ~Base()`）→ **动态绑定**
   - ① 先调用**派生类析构函数** ~Derived() → 输出 **~Derived()**
   - ② 自动调用**基类析构函数** ~Base() → 输出 **~Base()**

**与4.4对比的关键区别：**
- 4.4的析构函数**不是虚函数** → 只调用了基类析构（派生类部分未清理！）
- 4.8的析构函数**是虚函数** → 先调用派生类析构，再调用基类析构（完全清理！）

**核心结论：** 如果基类指针可能指向派生类对象（多态场景），基类析构函数必须声明为virtual，否则会导致派生类资源泄漏。
